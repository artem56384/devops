# Ansible

## Краткое описание

Ansible — это инструмент автоматизации, предназначенный для управления конфигурацией, развертывания приложений, оркестрации и выполнения административных задач на удалённых узлах.

Ansible работает по принципу agentless, то есть не требует установки отдельного агента на управляемые серверы. Для подключения обычно используется SSH.

## Основные возможности Ansible

Ansible позволяет:

* автоматизировать настройку серверов
* устанавливать и обновлять пакеты
* управлять пользователями, группами и правами доступа
* копировать и изменять файлы конфигурации
* управлять сервисами
* выполнять команды и скрипты на удалённых хостах
* развертывать приложения
* описывать инфраструктуру в виде кода

## Основные компоненты Ansible

### Inventory

Inventory — это список управляемых узлов.
Он может быть статическим или динамическим.

В inventory указываются:

* хосты
* группы хостов
* переменные для узлов и групп

### Playbook

Playbook — это файл с описанием последовательности задач, которые должны быть выполнены на удалённых хостах.

Playbook пишется в формате YAML.

### Task

Task — это отдельная операция в playbook, например:

* установка пакета
* создание пользователя
* копирование файла
* запуск сервиса

### Module

Module — это встроенный функциональный блок, который выполняет конкретное действие.

Примеры модулей:

* apt
* yum
* service
* copy
* file
* user
* ping

### Role

Role — это способ структурировать playbook и разложить конфигурацию по каталогам.

Обычно role включает:

* tasks
* handlers
* templates
* files
* vars
* defaults

### Handler

Handler — это специальная задача, которая выполняется только при изменении состояния, например после обновления конфигурации сервиса.

### Variables

Variables используются для хранения настраиваемых значений, чтобы playbook был более гибким и переиспользуемым.

### Templates

Templates позволяют генерировать конфигурационные файлы на основе шаблонов с подстановкой переменных.

## Best Practices по написанию playbook

### 1. Делать playbook идемпотентными

Playbook должен давать одинаковый результат при повторном запуске и не выполнять лишние действия, если система уже находится в нужном состоянии.

### 2. Использовать roles

Для крупных проектов лучше разбивать логику по ролям. Это улучшает читаемость, повторное использование и поддержку кода.

### 3. Давать понятные имена задачам

У каждой задачи должно быть понятное имя, чтобы было легко читать вывод выполнения playbook и понимать назначение задачи.

### 4. Не хранить чувствительные данные в открытом виде

Пароли, ключи и другие секреты не следует хранить в playbook в открытом виде. Для этого рекомендуется использовать механизмы шифрования, например Ansible Vault.

### 5. Использовать переменные вместо жёстко заданных значений

Повторяющиеся значения лучше выносить в переменные. Это упрощает поддержку и делает playbook более универсальным.

### 6. Разделять окружения

Для разных окружений, например development, test и production, лучше использовать отдельные inventory и переменные.

### 7. Минимизировать использование shell и command

Если есть специализированный модуль Ansible, лучше использовать его вместо shell или command, так как модули обычно более безопасны и идемпотентны.

### 8. Использовать handlers для перезапуска сервисов

Перезапуск сервиса лучше выполнять только тогда, когда действительно изменился конфигурационный файл.

### 9. Соблюдать аккуратную структуру проекта

Файлы playbook, inventory, roles, templates и vars должны быть организованы логично и предсказуемо.

### 10. Проверять playbook перед применением

Перед запуском рекомендуется использовать проверку синтаксиса и по возможности тестировать playbook на отдельном стенде.

## Вывод

Ansible является удобным инструментом автоматизации инфраструктуры и администрирования серверов.
Его основными преимуществами являются простота, отсутствие агентов, читаемый формат YAML и возможность описывать инфраструктуру в виде кода.
Соблюдение best practices делает playbook более понятными, безопасными и удобными в сопровождении.
